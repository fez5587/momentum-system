# Momentum consolidation plan

_Scope-only plan produced 2026-06-23 by a read-only 4-agent mapping workflow (wf_3fb5d523). No code was changed. Items are for operator approval; execute in P0→P2 order._

## TL;DR

The single-Postgres migration is functionally COMPLETE at the connection layer (all live code flows through storage/db.py -> db_pg -> one DATABASE_URL Postgres; no live duckdb.connect exists), so the datastore work is cleanup, not architecture. The genuine open architectural item is the two entry chokepoints — but the maps are slightly stale in our favor: the VWAP gate (commit e3ac89c) and daily-entry cap (ac1ac50) are now ALSO on the slow/auto path, so the remaining fast-only gaps are narrower than stated: extension, day-extension, halt, liquidity sizing, and rank-concentration. The whole entry-path effort hinges on one unverified empirical claim — whether live fills originate from the fast/trigger path (which has all gates) or the slow auto path (which lacks the five above) — so the plan opens by PROVING that via the existing approved_by tag ('auto' vs 'orb_trigger'), since it determines whether those gaps are live or theoretical. Sequencing favors reversibility: P0 = the empirical fill-source check plus the zero-risk, zero-blast-radius wins (delete the untracked gitignored archive/ tree and 232MB zip; drop the orphaned mem_ test schema confirming the known PG-isolation landmine and close its recurrence). P1 = behaviour-neutral cleanups (delete the dead core/ and telemetry/ packages; remove vestigial .duckdb path-strings so a file backend can't be reintroduced; archive the two frozen pre-cutover .duckdb files AFTER a pg_dump; extract the fast path into a single _place_entry helper as a pure refactor). P2 = the higher-blast changes done last and behind a flag (enrich the signal_ready snapshot with live price/day_open/volume/rank — the structural root cause of the dead slow-path gates; route approve_order through the unified helper and choose ONE fill model; the Schwab delete-vs-repair product decision; and verifying the symbols/telemetry_events empty tables aren't silently-dead writers). Throughout, the non-negotiables are: keep every shared-state mutation (self._fresh_entries, self._armed, self._requested_symbols, self._exited_today, self._cooldown_until) inside @_locked methods, never touch the naked-stop protections in expire_stale_entries, preserve the auto-vs-manual approval exemption, and never block an entry on missing data (fail-open like the existing above_vwap handling). Key files: trading_execution.py (entry chokepoints + shared state), run_live_paper.py (fast-path call site + boot warning), storage/projections.py (ready-signal snapshot), research/schema.py + research/multi_schema.py (vestigial duckdb labels), storage/db_pg.py + tests/conftest.py (mem_ schema isolation), storage/pg_schema.sql (orphaned tables).


> **Safety note (answered):** the P0 'which path produces fills' question is already settled — on 2026-06-23 **47/47 live entries were `approved_by="auto"` (the slow path)**. So the fast-path-only gates the map flags as missing on the live path — **extension / day-extension / halt-guard / liquidity-sizing / rank** — are a REAL current gap, not theoretical. (VWAP gate + daily-cap are now on both paths as of today's commits.)

### P2 — Route approve_order through the unified _place_entry and pick ONE fill model behind a flag
- **blast radius:** high  |  **effort:** large
- **action:** After the snapshot is enriched (prior step), make approve_order (trading_execution.py:740) build a ctx from the enriched approval payload and call the SAME _place_entry instead of its inline executor.execute, so both paths share one gate set. Then decide the single fill model (resting limit at signal entry, slow :713, vs marketable limit a hair above trigger, fast :929-931) and gate the change behind an env flag so it can be A/B'd against current behaviour and rolled back. Only after production logs confirm fills flow exclusively through _place_entry, delete the divergent inline blocks and the boot WARNING (run_live_paper.py:812-815). Note: VWAP gate (e3ac89c) and daily-cap (ac1ac50) are ALREADY on the auto path, so those two no longer need plumbing — the remaining gaps are extension, day-extension, halt, liquidity sizing, and rank.
- **why:** This is the genuine architectural win — one entry chokepoint with one gate set — but it is the highest-risk change in the plan: it alters how live orders are placed and changes fill economics (resting vs marketable limit). It must come last, behind a flag, after the helper exists and the snapshot carries the needed context, and after P0 has shown where fills actually originate. The flag + A/B + 'delete only after production confirmation' sequencing is what keeps 'don't break what we have' true. Honest risk: get the shared-state locking or the manual-override exemption wrong and you either double-count the daily cap or strip a protection.
